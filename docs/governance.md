## 2) Политика данных

### 2.1 Категории данных
- **Obsidian Vault**: исходные заметки пользователя (в т.ч. потенциально персональные данные).
- **Markdown отчёты**: развернутые результаты квизов и собеседований, удобные для чтения и повторного анализа LLM.
- **Metrics DB**: численные метрики, агрегаты, прогресс по темам/тегам для визуализации в UI.

### 2.2 Принципы
- По умолчанию — **минимизация**: не хранить лишнее, не логировать контент заметок.
- Любая отправка контента заметок во внешнюю LLM — **явно осознанная** (конфигурация и UX подтверждения).
- Vault предполагается под git: rollback возможен, но это не заменяет подтверждения и ограничители.

## 3) Политика логов

### 3.1 Что логируем
- технические события: вызовы инструментов, длительности, коды ошибок, id сессий;
- агрегаты: количество найденных заметок, размеры материалов, токены, число вопросов квиза;
- результаты безопасности: факт запроса write, факт подтверждения/отклонения пользователем.

### 3.2 Что НЕ логируем
- полное содержимое заметок;
- секреты (API keys, JWT, пароли);
- полные промпты, содержащие пользовательский контент (без режима отладки и отдельного согласия).

## 4) Защиты и политики выполнения

### 4.1 Write-операции в Obsidian vault
Правило: **write только после explicit confirm**.

Рекомендуемый протокол:
1) агент строит план изменений;
2) показывает diff/preview;
3) пользователь подтверждает;
4) выполняется патч;
5) выполняется проверка валидности YAML frontmatter.

### 4.2 Защита от prompt injection
- заметки и web-страницы считаем **untrusted input**;
- контент источников не может менять системные политики;
- любые опасные действия (write, запуск кода, массовые операции) требуют подтверждения и ограничений.

### 4.3 Политика интервьюера
- интервьюер **не выдаёт** правильные ответы и готовый код;
- интервьюер даёт подсказки, указывает на ошибки, просит уточнения и проверяет понимание;
- допускается обсуждение решений после завершения попытки пользователя или по отдельному режиму.

### 4.4 Безопасность Algo Sandbox
- выполнение кода в изолированном контейнере;
- лимиты CPU/RAM, таймауты;
- запрет сети;
- минимальные монтирования файлов;
- логирование только технических результатов (статус тестов, время выполнения), без утечки пользовательского кода в внешние сервисы.

## 5) HSM как контроль изменений стека

HSM считается источником истины состава стека:
- любые изменения делаются через intent-команды HSM;
- после изменений обязателен `hsm sync`;
- перед sync рекомендуется `hsm check`.
