# Спецификация: Retriever

## Назначение
`Retrieval Service` отвечает за подготовку воспроизводимого контекста для сценариев работы с заметками, квизами и алгоритмическими интервью. Сервис преобразует разнородные источники в нормализованные записи индекса, выполняет поиск и возвращает [`RetrievalSnapshot`](../system-design.md), пригодный для использования оркестратором.

## Границы модуля
Retriever отвечает за:
- ingestion и нормализацию заметок, тегов, отчётов и банка задач;
- инкрементальное обновление индекса;
- поиск по тегам, метаданным и текстовым признакам;
- при необходимости reranking;
- формирование snapshot с provenance и оценками релевантности.

Retriever не отвечает за:
- выбор бизнес-сценария;
- генерацию prompt и управление LLM;
- любые side effects в пользовательских артефактах;
- интерпретацию retrieval-результатов как окончательного пользовательского ответа.

## Источники данных

| Источник | Назначение | Каноничность |
|---|---|---|
| Obsidian notes | основной материал для tagging, augmentation и quiz | канонический источник содержимого |
| YAML frontmatter tags | теги для поиска и навигации | канонический источник тегов |
| Метаданные заметок | путь, время изменения, идентификаторы | канонический источник служебных атрибутов |
| Fixed problem bank | задачи, теги, тестовые ресурсы | канонический источник interview задач |
| Markdown reports | история прогресса и обратной связи | вторичный источник для mentor и recap |
| Web search results | внешние материалы для augmentation | временный источник до подтверждения записи |

## Основные контракты

### `RetrievalQuery`
Минимальный контракт поискового запроса:
- `query_id`
- `scenario`
- `scope`
- `text_query`
- `tag_filters`
- `metadata_filters`
- `top_k`
- `rerank_policy`
- `source_policy`

### `RetrievedItem`
Единица найденного материала:
- `item_id`
- `source_type`
- `source_ref`
- `content_ref`
- `rank_score`
- `metadata`
- `provenance`

### `RetrievalSnapshot`
Неподвижный пакет контекста для одного шага:
- `snapshot_id`
- `query_ref`
- `items`
- `created_at`
- `budget_hint`
- `policy_flags`

## Пайплайн

### 1. Ingestion
- чтение заметок и метаданных;
- извлечение YAML frontmatter;
- нормализация тегов;
- чтение fixed problem bank;
- подключение внешних web results как временного источника.

### 2. Нормализация
- приведение тегов к каноническому виду;
- выделение текстовых фрагментов для поиска;
- очистка дубликатов;
- маркировка untrusted источников;
- присвоение стабильных `source_ref`.

### 3. Индексация
- инкрементальное обновление по изменившимся документам;
- раздельный учёт note index, report index и problem bank index;
- сохранение версий snapshot-friendly metadata.

### 4. Поиск
- фильтрация по сценарию;
- поиск по тегам и метаданным;
- при необходимости текстовый поиск по фрагментам;
- ограничение размера выдачи в соответствии с context budget.

### 5. Reranking
- опциональный шаг;
- включается только если baseline retrieval недостаточен;
- не должен менять provenance и source refs;
- может быть отключён без изменения внешнего контракта retriever.

### 6. Материализация snapshot
- сбор выбранных элементов;
- фиксация rank scores и фильтров;
- запись snapshot id для audit и повторного использования в отчётах.

## Сценарные политики поиска

| Сценарий | Приоритетные источники | Ограничения |
|---|---|---|
| Tagging | заметка, tag index, соседние заметки по тегам | не нужен доступ к web results |
| Augmentation | заметка, web results, существующие ссылки | web sources считаются временными |
| Quiz | заметки по тегам, прошлые отчёты по теме | не нужен доступ к solution bank |
| Interview | fixed problem bank, история попыток, progress summaries | запрещено подтягивать готовые решения |
| Mentor | отчёты, summaries, progress artifacts | не запускает sandbox и не пишет в vault |

## Политики качества
- YAML frontmatter является источником истины для тегов.
- Inline tags в теле заметки не используются как канонический индексный сигнал.
- Каждый `RetrievalSnapshot` должен содержать provenance.
- Большие документы должны резаться на фрагменты, пригодные для context budget.
- Результаты поиска не должны скрыто подтягивать материалы, запрещённые для текущей роли.

## Ошибки и деградация

| Код | Смысл | Поведение |
|---|---|---|
| `RET_INDEX_STALE` | индекс устарел | запустить reconcile или reindex |
| `RET_NOT_FOUND` | релевантные материалы не найдены | вернуть пустой snapshot с пояснением |
| `RET_POLICY_BLOCKED` | источник запрещён для роли или сценария | исключить источник и зафиксировать telemetry |
| `RET_BUDGET_EXCEEDED` | выдача превышает бюджет | сократить top_k или применить summarization |
| `RET_EXTERNAL_UNAVAILABLE` | внешний источник недоступен | продолжить по локальным данным |

## Наблюдаемость
Retriever обязан публиковать:
- `query_id`, `scenario`, `scope`
- число найденных элементов до и после фильтрации
- latency по этапам поиска
- число элементов, исключённых policy-фильтрами
- размер итогового snapshot
- факт использования reranking

Полный контур метрик и evals описан в [`docs/specs/observability-evals.md`](observability-evals.md).

## Критерии готовности реализации
- поиск по тегам и метаданным воспроизводим и не зависит от провайдера LLM;
- snapshot содержит provenance и стабилен для последующего аудита;
- деградация внешних источников не ломает локальные сценарии;
- retrieval не нарушает role-scoped ограничения;
- contracts остаются совместимыми с [`docs/specs/orchestrator.md`](orchestrator.md) и [`docs/specs/memory-context.md`](memory-context.md).
